INSTRUCTIONS FOR AI REPORT GENERATION

*** PRIORITY ACTIONS FORMAT - READ THIS FIRST***

In the PRIORITY ACTIONS section, you MUST insert the literal HTML tag <br> between the action and "Expected Impact:"

CORRECT: <p>• Fix broken links.<br>Expected Impact: Reduce 404 errors</p>
WRONG:   <p>• Fix broken links.Expected Impact: Reduce 404 errors</p>
WRONG:   <p>• Fix broken links. Expected Impact: Reduce 404 errors</p>

The literal characters .<br>Expected must appear — period, then br tag, then Expected. No space between period and br tag.

[!] CRITICAL: USE ONLY REAL DATA FROM TOOL RESPONSES

This template contains NO actual data - only structure and formatting guidelines. ALL metrics, numbers, percentages, and values MUST come from the tool responses and dashboard data provided in the SOURCE MATERIAL section. DO NOT use any placeholder values, example numbers, or hypothetical data.

[!] FORMAT REQUIREMENT - USE SEMANTIC HTML:

  • <h4> for main section headings, <b> for subsection labels
  • Wrap ALL text in <p> tags, use bullet character (•) for lists
  • Include specific numbers and metrics from tool responses in your text
  • NO emojis or progress bars - keep report easy to scan with clear structure

FACET LINKING REQUIREMENT - MAKE FINDINGS CLICKABLE:

  • When mentioning specific facet findings, wrap them in <span> tags with data attributes
  • Simple facet format: <span data-facet="facetName" data-facet-value="facetValue">finding text</span>
  • Nested facet format (REQUIRED for .source/.target facets): <span data-facet="checkpoint" data-facet-value="parentCheckpoint" data-nested-facet="checkpoint.source" data-nested-value="specificValue">finding text</span>

[!] CRITICAL: For Any Facet Ending in .source or .target

  • ALWAYS include BOTH the parent checkpoint AND the nested facet
  • [X] WRONG: data-facet="error.source" data-facet-value="focus-loss"
  • [OK] CORRECT: data-facet="checkpoint" data-facet-value="error" data-nested-facet="error.source" data-nested-value="focus-loss"
  • This generates URL: &checkpoint=error&error.source=focus-loss

[!] CRITICAL: Use ACTUAL Facet Values, NOT Display Text

  • ALWAYS use the exact facet value from the tool results, not human-readable text
  • [X] WRONG: data-nested-value="All paid traffic" (display text)
  • [OK] CORRECT: data-nested-value="paid" (actual facet value)
  • Example: If tool shows {"value": "paid", "count": 1234}, use "paid"
  • Example: If tool shows {"value": "focus-loss", "count": 567}, use "focus-loss"
  • The display text goes INSIDE the span, the actual value goes in data attributes

Correct vs Incorrect Examples:

  • [X] <span data-facet="checkpoint" data-facet-value="acquisition" data-nested-facet="acquisition.source" data-nested-value="All paid traffic">paid traffic</span>
  • [OK] <span data-facet="checkpoint" data-facet-value="acquisition" data-nested-facet="acquisition.source" data-nested-value="paid">All paid traffic</span>

Examples by Facet Type:

  • Simple checkpoint: <span data-facet="checkpoint" data-facet-value="click">click events show strong engagement</span>
  • Simple URL: <span data-facet="url" data-facet-value="/products">/products page has high bounce rate</span>
  • Simple device: <span data-facet="userAgent" data-facet-value="mobile">mobile users experience slower LCP</span>
  • Nested LCP source: <span data-facet="checkpoint" data-facet-value="cwv-lcp" data-nested-facet="cwv-lcp.source" data-nested-value="img">LCP images causing delays</span>
  • Nested error source: <span data-facet="checkpoint" data-facet-value="error" data-nested-facet="error.source" data-nested-value="focus-loss">focus-loss errors</span>
  • Nested click target: <span data-facet="checkpoint" data-facet-value="click" data-nested-facet="click.target" data-nested-value="button">button clicks</span>

General Rules:

  • Use facet names exactly as they appear in the tool results
  • For nested facets, ALWAYS set data-facet="checkpoint" with the parent checkpoint name
  • Then add data-nested-facet and data-nested-value for the specific source/target
  • Wrap entire finding statements, not just individual words
  • This enables readers to click findings and navigate directly to the relevant dashboard view with correct filters

[!] CRITICAL: Only Link to VERIFIED Facet Data

  • DO NOT create facet links unless the data exists as a selectable checkbox in the sidebar
  • Before adding a facet link, verify the value appears in the tool results with actual counts
  • If a facet/value combination doesn't have a corresponding checkbox, the link will be broken
  • Example of INVALID link: checkpoint=enter&enter.source=https://www.google.com/ — if this doesn't appear as a checkbox in sidebar, DON'T link it
  • Test: Would clicking this link check a box in the facet sidebar? If NO, don't create the link.
  • It's better to mention data WITHOUT a facet link than to create a broken/misleading link
  • Only wrap text in facet spans when you're CERTAIN the facet value exists in the sidebar data

[!] CORE STRUCTURE - COMPREHENSIVE DATA COVERAGE:

  • BE SPECIFIC: Use exact numbers, not vague summaries. "Mobile LCP is 3.2s vs desktop 1.8s" not "performance varies"
  • INCLUDE BREAKDOWNS: Show mobile vs desktop, top vs bottom pages, source comparisons
  • NAME SPECIFICS: List actual page URLs, error messages, referrer domains
  • SHOW THRESHOLDS: Compare to good/poor thresholds (LCP: 2.5s, CLS: 0.1, INP: 200ms)
  • NO REPETITION: Each insight appears ONCE in the most relevant section. Don't repeat the same metric/finding across multiple sections.
  • Target: 1200-1600 words - concise but specific

NARRATIVE STYLE - TELL A STORY WITH DATA:

[!] Your report should read like a story, not a data dump. Lead with what matters, explain why it matters, and connect insights across sections.

1. HOOK THE READER:

  • Open the Executive Summary with the most surprising or important finding
  • Good: "Mobile users are abandoning the checkout page at 3x the rate of desktop—and a 4.2s LCP is the likely culprit."
  • Bad: "This report analyzes the website performance for the last 30 days."

2. HIGHLIGHT SURPRISES:

  • Call out counterintuitive findings: "Despite strong overall traffic, organic search converts 40% worse than paid."
  • Note unexpected patterns: "The blog drives 60% of traffic but only 5% of conversions."
  • Flag anomalies: "Error rates spiked 300% on mobile Safari specifically."

3. CONNECT THE DOTS:

  • Link causes to effects: "The slow LCP on mobile (3.2s) likely explains the 65% mobile bounce rate."
  • Show relationships: "Germany shows high traffic but poor engagement—latency issues may be the cause."
  • Build a narrative: Each section should flow into the next, not feel isolated.

4. END SECTIONS WITH "SO WHAT":

  • After presenting data, explain the implication: "This means roughly 12,000 users per month may be leaving before the page loads."
  • Quantify impact where possible: "Fixing this could recover an estimated $X in conversions."

5. BE SPECIFIC BUT ENGAGING:

  • Include ALL metrics from tool responses—but weave them into a narrative
  • Use comparisons: "Mobile LCP at 3.2s is nearly double desktop's 1.8s"
  • Add context: "This exceeds Google's 2.5s threshold, hurting SEO rankings"

Example of BORING (data dump):

"LCP is 2.3s. CLS is 0.1. INP is 180ms. Mobile is slower. Desktop is faster. Bounce rate is 45%."

Example of ENGAGING (narrative with same data):

"Here's the story: desktop users enjoy a snappy 1.8s LCP, but mobile users wait nearly twice as long at 3.1s—crossing Google's 2.5s threshold. That extra wait shows up in the numbers: mobile bounce rate hits 58% versus desktop's 32%. The culprit? A hero image on the homepage accounts for 56k slow LCP measurements. Fix that one image, and you're looking at a potential 20% bounce reduction."

[!] The engaging example: hooks with insight, shows cause-effect, names the specific problem, and quantifies the opportunity. This is the standard.

6. TEXT FORMATTING - USE SPARINGLY:

  • Maximum 1 bold phrase per section - use only for the single most critical insight or metric
  • Maximum 1 italic phrase per section - use only for emphasis on a surprising finding or key implication
  • Overusing bold/italic dilutes impact - when everything is emphasized, nothing is
  • Let the data speak for itself; formatting should highlight, not decorate
  • Exception: Section headings (h4) use bold as standard

Number Formatting - USE <number-format> TAG FOR ALL NUMBERS:

[!] CRITICAL: Wrap EVERY number in the report with <number-format> tags. This includes metrics, counts, percentages, durations, error counts, session counts, page views, etc.

Format:

<number-format title="[raw value] [trend text]" trend="[rising|falling|stable]" sample-size="[count]">
  <span class="formatted-value">[formatted display]</span>
</number-format>

Attributes:

  • title: Full raw value with uncertainty and trend if available (e.g., "227129980 ±5123047 and falling"). If no uncertainty/trend data, just use raw value (e.g., "4500")
  • trend: One of "rising", "falling", or "stable". Use "stable" if trend is unknown.
  • sample-size: Number of samples/sessions used for the metric. If unknown, use approximate or omit the attribute.
  • formatted-value: Human-readable shortened version (e.g., "230m", "2.3s", "45%", "4.5k")

Examples:

  • Large counts: <number-format title="227129980 ±5123047 and falling" trend="falling" sample-size="7551"><span class="formatted-value">230m</span></number-format> page views
  • Durations: <number-format title="2.34s ±0.12s and rising" trend="rising" sample-size="4500"><span class="formatted-value">2.3s</span></number-format> LCP
  • Percentages: <number-format title="45.2% ±2.1%" trend="stable" sample-size="890"><span class="formatted-value">45%</span></number-format> bounce rate
  • Error counts: <number-format title="1234 errors" trend="stable" sample-size="890"><span class="formatted-value">1.2k</span></number-format> errors
  • Small counts: <number-format title="23 sessions" trend="stable"><span class="formatted-value">23</span></number-format> mobile sessions

[!] DO NOT write bare numbers:

  • [X] WRONG: "Mobile users experienced 4,900 sessions"
  • [OK] CORRECT: "Mobile users experienced <number-format title="4900 sessions" trend="stable"><span class="formatted-value">4.9k</span></number-format> sessions"

[!] Core Web Vitals Interpretation Guidelines

When analyzing LCP, CLS, and INP metrics:

  • If metric titles contain "and rising" or "and falling", these indicate TRENDS, not absolute performance
  • "Rising" means the metric is getting worse over time, "falling" means it's improving
  • DO NOT conclude the site is bad based solely on "rising" trends - consider the actual metric values
  • Focus on the current metric values and their impact, not just the trend direction
  • A metric can be "rising" but still within acceptable performance thresholds

Facet Analysis Guidelines

For ALL facets being analyzed:

  • ALWAYS check the title field of each facet (found in the help link's title attribute) to understand what it represents and its purpose
  • Use these descriptive titles to provide context and meaning to your insights and recommendations
  • DO NOT focus primarily on performance metrics - provide balanced analysis across ALL available facets
  • Include insights from user behavior, traffic acquisition, content engagement, errors, and other non-performance facets
  • Each facet provides unique business value - ensure comprehensive coverage of all data types

[!] UTM TRACKING - PRIVACY BY DESIGN:

  • UTM data only records parameter names (utm_source, utm_medium, etc.), NOT their values — this is intentional for privacy
  • DO NOT recommend "capturing UTM values" or suggest this is a limitation to fix
  • DO NOT say "only parameter names are recorded" as if it's a problem

  • Valid insights: "UTM parameters are present on X% of traffic" or "utm_source is the most commonly used parameter"

[!] CRITICAL: INTERPRETING CSS CLASSES FOR METRIC SEVERITY

When analyzing tool results, metrics include CSS classes that indicate severity levels. You MUST pay special attention to these class indicators:

Performance Metrics (LCP, CLS, INP):

  • classes: "performance score-poor" = CRITICAL ISSUE - Metric is failing Core Web Vitals standards. This directly impacts user experience and SEO rankings. MUST be highlighted prominently.
  • classes: "performance score-ni" = Needs Improvement - Metric is borderline. Mention if relevant but not critical.
  • classes: "performance score-good" = Good - Metric is passing. Can be mentioned as a positive.

Business Metrics (Engagement, Conversion, Bounces, Acquisition):

  • classes: "business-metric engagement conversion" - Engagement/Conversion metric. If value is below 25%, this is a CRITICAL ISSUE indicating serious user interaction problems. Values above 75% are healthy.
  • classes: "business-metric acquisition bounces" - Bounce rate metric. If value is above 60%, this is a CRITICAL ISSUE indicating serious retention problems. Values below 40% are healthy.
  • classes: "business-metric acquisition organic" - Organic traffic/acquisition metric. Low values may indicate SEO or discoverability issues.
  • Note: Metric values may have <meter> attributes (low, high, optimum) that provide threshold guidance

[!] MANDATORY: When you see metrics with "score-poor" or critical thresholds, you MUST:

  • Prominently mention them in the relevant sections with specific values and context
  • Include them in "JAVASCRIPT ERRORS THAT NEED ATTENTION" if they relate to technical/performance issues AND errors ≥1% of traffic
  • List them in "PRIORITY ACTIONS" with clear, actionable remediation steps
  • Reference them in "EXECUTIVE SUMMARY" if they have significant business impact
